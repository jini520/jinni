# 워크플로우 제어 규칙

## 코드 수정 절대 승인 원칙 (최우선)

**사용자의 명시적 승인 없이는 어떤 코드도 절대 수정하지 않는다.** 아래 구현 승인 게이트의 "즉시 실행 가능" 예외보다 이 원칙이 우선한다.

- 시안·분석·제안에 대한 피드백과 논의는 코드 수정 승인이 **아니다**.
- "~로 하자", "~가 좋아", "B안으로" 같은 **선택/방향 표현도 수정 승인이 아니다** — 내용 확정과 코드 적용 승인은 별개다. 확정 후 "이대로 코드에 적용할까요?"를 묻고 승인을 기다린다.
- 승인은 "구현해", "진행해", "적용해", "GO", "수정해줘" 등 **코드 변경을 직접 지시하는 표현**으로만 성립한다.
- 승인 범위는 그 시점에 합의된 변경에 한정된다. 새 변경이 생기면 다시 승인받는다.

## 구현 승인 게이트

비자명한 작업(3단계 이상 또는 아키텍처 결정)에서는
반드시 다음 순서를 따를 것:

1. 코드베이스 탐색 및 현황 파악
2. 접근법 2-3가지와 트레이드오프 제시
3. **"구현해"라는 명시적 승인을 기다릴 것**
4. 승인 후 구현 시작

다음 상황에서 즉시 실행 가능:

- 명확한 버그 수정 (증상이 분명한 경우)
- 1-2단계 이하의 단순 수정
- 사용자가 "바로 구현해" 또는 "GO"를 말한 경우

승인 키워드: "구현해", "진행해", "GO", "go", "ok 해줘"
대기 키워드: 질문, "어때?", "어떻게 생각해?", "검토해줘"

## 계획 우선 원칙

모호한 요청을 받으면 편집하기 전에 질문할 것:

- "무엇을 달성하려는지 명확하지 않습니다. [A]를 원하시나요, [B]를 원하시나요?"
- 단순히 더 많은 일을 하는 것보다 명확화가 우선
- 복잡한 작업은 tasks/todo.md에 계획을 먼저 작성
